## Features

### Next sprint

- **Plan-vs-reality as a first-class finding lens (`agent.expectation_match.*`)** — When an agent is handed a goal ("order a pizza on this webapp"), it forms an internal game plan BEFORE any tool call: expected route patterns, expected affordances on the home page, expected step count, anticipated friction points, where it thinks the cart icon will live. Today that plan dies in the agent's first turn — never written down, never compared to reality. Capture it. The proposal: (1) Add a `plan_capture` phase to the walk prompt — before the first `browser_navigate`, the agent emits a structured `prior_plan` (yaml: expected_route_pattern, expected_affordances, expected_step_count, anticipated_friction, affordance_assumptions, archetype_assumed). Store on `runs.prior_plan jsonb`. (2) On each step, the agent emits a one-token verdict: `match | extension | surprise | deviation` against its plan, plus a one-line "what got revised." Store in `run_steps.plan_delta jsonb`. (3) New finding heuristic family `agent.expectation_match.{route|affordance|copy|step_count|friction}` — every `deviation` and `surprise` is a candidate finding (severity by category — a missing "Add to cart" button on the product page is critical, an unexpected popup-newsletter modal is minor). (4) Personas grow `prior_archetype` (`"doordash-style food delivery"`, `"single-restaurant direct"`, `"shopify-style commerce"`) so the same flow walked under different archetypes generates different expectation sets — that diff IS the data. (5) Flow YAML grows optional `prior_overrides` so authors can stipulate "this isn't shopify-style, agents shouldn't expect a cart icon at top-right." (6) Reflection tab gets a "Plan vs reality" section: prior_plan on left, observed reality on right, deltas highlighted. This is the wedge no deterministic test framework can build — it only works because Rove's apparatus is an agent with priors, not a script with assertions. Roughly 1-2 days of prompt + schema + UI work. **This is probably the single highest-leverage thing on the roadmap; sequence it before most other Phase D items.**
- **Add `browser_press_key` to the accessibility + agent persona toolset** — Built-in `accessibility` persona claims keyboard-only and reads off WCAG 2.2 in its `promptAddendum`, but in practice uses `browser_click`. Every WCAG 2.1.1/2.1.2/2.4.3/2.4.7 finding it emits today is on hypothesis, not observation — the persona never tried the keyboard. Adding `press_key` is the only candidate in the unused Playwright tool set that is **adversarial-by-nature** (generates findings) rather than convenience-by-nature (smooths flake). A keyboard-only walk on a modern web app fails constantly: modal traps focus on the wrong element, tab order skips a critical field, custom dropdown doesn't respond to space, skip-link goes nowhere because the target lost its `id`. None surface from a mouse walk; all surface the moment something presses Tab. Build: expose `browser_press_key` to the walk; branch the prompt by persona (`accessibility` required to use it, `agent` personas use it to probe `agent.no_hover_only` keyboard equivalents, human personas don't get the nudge); add `expected_keyboard_navigation` to flow YAML so authors can assert "Tab from email lands on password, not privacy-policy link"; auto-file a finding when focus disappears off-document or tab order skips a stated step. Explicitly DO NOT make this a general-convenience tool — it earns its place by being a *constraint* for the keyboard persona, not a shortcut for everyone. Small change, large credibility lift — our `.claude/rules/personas-and-flows.md` claim of "WCAG 2.2 AA coverage" becomes a fact instead of marketing.
- **Native browser dialogs as first-class run artifacts** — Every test framework treats `alert()`/`confirm()`/`prompt()`/`beforeunload` as a thing to suppress so the test passes. Rove should treat them as a thing to *record* so the readiness story is honest. Today the MCP proxy doesn't intercept dialog events at all; Playwright's default auto-dismisses them in milliseconds and the walk never knows they fired. That kills three things: (1) `agent` personas never get a finding when a destructive action is gated by `confirm()` that they literally cannot perceive via the accessibility tree; (2) flow authors can't write `success_predicate` clauses that assert "delete-confirm dialog appears with text matching /Delete this property\?/"; (3) the reflection tab can't surface "walk blocked by N native dialogs" as a surprise. Build: proxy registers `page.on("dialog", …)`, writes a `run_dialogs` row (or `run_steps` of `kind="native_dialog"`) with type/message/default_value/triggered_by_tool, auto-files a finding gated on persona category (`agent.accessibility_tree_completeness` for agent personas; Nielsen "user control and freedom" + WCAG copy-clarity for humans only when copy/severity warrants), then dismisses with a per-persona policy (`Persona.constraints.native_dialog_policy = perceive_and_act | perceive_blind | dismiss_silently`). Add `expectations.native_dialog` to flow YAML so authors can assert a dialog *must* fire before a step. UI: chip on the triggering filmstrip step ("fired `confirm()`"), reflection-tab surprise category, auto-filed findings tagged `auto:dialog` to distinguish from agent-reasoned findings. Minimal MCP-prompt change is one line: "if your tool calls stop affecting the page, call `browser_handle_dialog` to query — the page may be blocked by a browser-native dialog." Roughly a half-day of proxy + migration + persona-constraint + small UI work.

### Backlog

- **Retire the install-bundled service-role key by routing sink writes through Wire** — Today (2026-05-14, alpha.15) the `/api/install/exchange` response ships the `ROVE_SUPABASE_SERVICE_ROLE_KEY` to every worker so the existing sink path (`getSupabaseClient`) just works. That regresses what `docs/plans/worker-tokens.md` was built to prevent: anyone who captures the 5-min install code now has 5 minutes to grab a service-role key, much bigger blast radius than just a worker JWT. Proper fix is the Wire-sink-relay proposal in `docs/proposals/wire-sink-relay.md`: worker emits `rove.sink.*` events to a local Wire instance → federated to a dashboard-side Wire peer → relay consumer with service-role does the actual Supabase writes. No Wire code changes needed. Sent to Fondant (Tim's agent) for review on 2026-05-14. **Do not ship this install flow to operators outside Agiterra until Wire-relay lands.**
- **Workspace-less walk execution for daemons installed via `/setup`** — Dogfooding the install flow on 2026-05-14 exposed that the daemon-spawned `rove run` subprocess calls `resolveWorkspace()` → `loadRoveConfig()`, which walks up from `/` (launchd's cwd) and throws because there's no `rove.config.ts` (and no repo checkout exists — that's the whole point of one-paste install). Three pieces needed to fully close the loop: (1) `flows.yaml_body text` column so the flow spec lives in supabase, not just on disk; (2) `parseFlowFile` + sync writes the YAML body; (3) `rove run` accepts a `--project-id` flag and fetches flow+persona from supabase when no local workspace is found (synthesize a temp workspace at `~/.rove/run/<runId>/`). Personas are already shipped in `BUILT_IN_PERSONAS` in the CLI tarball; only flows need the fetch path. Daemon-spawn args in `walk.ts` also need `--project-id` and the right `cwd`. Until this lands, `/setup` installs work end-to-end (daemon comes online, claims jobs) but the spawned walk subprocess fails with "No rove.config found from /" — captured in finding `agent_jobs.id=9ebef158-b230-4f78-9fc4-2849668af031` error column. Plan-sized work, ~1-2 hours.
- **Finding lifecycle: "mark fixed" / snooze / suppress** — Dogfood walk 3 (`f4b19692`) re-flagged the `aria-hidden`-on-emoji finding even though production HTML had the fix. Non-deterministic personas will over-report; without lifecycle controls (snooze, mark-fixed, suppress-pattern) the signal will drown in repeat noise once we have more flows. Likely lives next to dedup. Findings IDs from walk 3 don't matter — the point is the UX gap.
- **Richer page titles for agent navigation context** — Walk 3 surfaced `<title>Sign in · Rove</title>` as too thin for an agent verifying it landed on the right surface. Cheap fix per route; nicer to do as a general pass. Phase D-2 polish, not urgent.
